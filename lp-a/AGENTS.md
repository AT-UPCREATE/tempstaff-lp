# AGENTS.md

このプロジェクトは、FigmaデザインをもとにしたWebサイト実装プロジェクトです。

目的は「見た目が綺麗で、あとから修正しやすいWebサイト」を作ることです。

## 最重要ルール

- このプロジェクトは途中まで実装済みです
- まず `README.md` を読んで、プロジェクトの前提を確認する
- その後、現在のファイル構成と既存コードを確認する
- いきなりコードを書かない
- 既存の命名規則・CSS設計・フォルダ構成を優先する
- 既存コードを勝手に大きく作り直さない
- 全面リファクタリングはしない
- 変更は小さな単位で行う
- 変更前に作業方針を簡潔に説明する
- 変更後に、変更したファイルと理由をまとめる

## 制作方針

- LPとしてではなく、実際のWebサイトとして保守しやすく作る
- SaaSや事業サイトに使える品質を目指す
- デザイン・余白・文字サイズ・レスポンシブを重視する
- コードは綺麗に分割する
- あとから人間が修正しやすい構造にする
- 似たパーツはコンポーネント化する
- 重複コードを書かない
- コメントは必要最低限にする
- わかりにくい箇所のみコメントを入れる
- 仮テキストではなく、可能な限り自然な日本語を入れる

## 技術方針

このプロジェクトは、以下のいずれかの構成で実装します。

- HTML / CSS / JavaScript
- React / Next.js

すでに実装済みの構成がある場合は、既存の技術構成を優先してください。

## ファイル構成ルール

基本構成は以下を想定します。

```txt
index.html
style.css
components/
sections/
assets/
README.md
AGENTS.md
```

ただし、すでに別の構成で実装されている場合は、既存構成を優先してください。

## CSS設計ルール

- 色、余白、フォントサイズはCSS変数で管理する
- 共通で使う値は `:root` にまとめる
- Figmaのpxをそのまま大量に固定値で書かない
- 余白や文字サイズは必要に応じて `clamp()` を使う
- PCだけでなくスマホ表示を必ず考慮する
- 375px / 768px / 1024px / 1440px を意識して調整する
- 同じような余白・文字サイズは統一する
- セクションごとにCSSを整理する
- 不要な上書きや場当たり的な `!important` は使わない

## HTML設計ルール

- 意味のあるHTMLタグを使う
- 意味のないdivを増やさない
- 見出し構造を正しくする
- `h1` は基本的にページ内で1つにする
- セクションごとに内容が伝わるclass名をつける
- SEOを意識した構造にする
- 画像には適切なaltを入れる
- 仮テキストではなく、可能な限り自然な日本語を入れる

## デザインルール

- Figmaの見た目をできるだけ再現する
- ただし、保守しにくいコードになる場合は整理して実装する
- 見出し、本文、ボタンの強弱を明確にする
- 1セクション1メッセージで整理する
- 角丸、影、背景色を使いすぎない
- デザインの統一感を崩さない
- 写真が入る前提でレイアウトを作る
- 画像は `assets/images/` に置く前提で扱う

## レスポンシブルール

- スマホ表示を必ず確認する
- 横並び要素はスマホで自然に縦積みにする
- 文字が小さくなりすぎないようにする
- 余白が詰まりすぎないようにする
- 画像が不自然に切れないようにする
- ボタンやリンクはタップしやすいサイズにする

## JavaScriptルール

- 不要なJavaScriptは書かない
- CSSで実現できるものはCSSで対応する
- 外部ライブラリを勝手に追加しない
- 既存のJSがある場合は、影響範囲を確認してから変更する
- 動きはシンプルで、保守しやすい実装にする

## Figma実装ルール

- Figmaのpxをそのまま固定値で大量に書かない
- 余白は共通変数、または `clamp()` で調整する
- 似た余白は統一する
- テキストサイズはデザインシステムに寄せる
- 背景画像は重要部分が切れないように調整する
- Figma上の装飾目的のグループは、HTMLではできるだけ簡潔に表現する
- 実装で迷う箇所は、仮対応として明記する

## 禁止事項

- その場しのぎのコードを書かない
- すべてを1ファイルに詰め込まない
- 既存コードを勝手に大きく作り直さない
- 意味のないdivを増やさない
- 重複コードを増やさない
- PCだけで成立するレイアウトにしない
- 架空の外部ライブラリを勝手に使わない
- デザインの統一感を崩さない
- 不要なコメントを大量に書かない
- `!important` を安易に使わない

## 作業手順

作業するときは、必ず以下の順番で進めてください。

1. `README.md` を読む
2. 現在のファイル構成を確認する
3. 既存コードの状態を確認する
4. できている部分・未実装部分・改善点を整理する
5. 変更方針を簡潔に説明する
6. 小さな単位で実装する
7. レスポンシブを調整する
8. 変更したファイルと理由をまとめる
9. 次に確認すべきポイントを提示する

## 途中実装プロジェクトでの注意

このプロジェクトは途中まで実装済みです。

- 既存の命名規則を優先する
- 既存のCSS設計を優先する
- 既存のフォルダ構成を優先する
- 改善が必要な場合も、小さな単位で変更する
- 全体を作り直す必要がある場合は、先に理由を説明する
- デザイン再現よりも、既存表示を壊さないことを優先する
- 未実装部分と修正すべき部分を分けて整理する

## 完了時の報告形式

```txt
【変更したファイル】
- ファイル名

【変更内容】
- 変更した内容

【確認してほしいこと】
- 確認ポイント
```
